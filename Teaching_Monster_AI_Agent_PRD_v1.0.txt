1. Product Overview
1.1 Purpose
This document specifies the full engineering requirements for the Teaching Monster AI Agent — an autonomous system that receives a JSON API request containing a course topic and student persona, and returns a complete educational video URL within 30 minutes, with zero human intervention.
The agent is designed to compete in and win the Teaching Monster Challenge (teaching.monster), a competition evaluating AI-generated teaching videos for secondary education subjects across physics, biology, computer science, and mathematics.


1.2 The core differentiator
Most competing teams will build video generators. The winning team builds a system that learns what better teaching means. The competition's Elo arena and AI Student evaluator are not just metrics — they are training signals. Every match is a data point for improving the agent's pedagogical policy.


Key insight: Video quality is table stakes. The differentiator is Pedagogical Content Knowledge (PCK) — the ability to understand how humans learn, not just what is factually correct. The agent must reason about scaffolding, misconceptions, cognitive load, and adaptive explanation style.

3. Five Capability Layers
The agent's intelligence is organised into five layers. Layers 1–2 define what the agent knows about learning. Layers 3–5 define how it improves over time. These layers inform the design of Modules M2–M5.


Layer 1 — Pedagogical intelligence
The agent must embody Pedagogical Content Knowledge (PCK): not just factual accuracy, but domain-specific knowledge of how to make content learnable. This layer powers Module M4 (script generation) and M5 (critic).


1a. Misconception detection
Before writing any script, the agent retrieves a domain-specific misconception library for the target concept. These are pre-curated, not generated, and cover the four competition subjects.
* Physics: force vs. momentum, weight vs. mass, current vs. voltage
* Biology: evolution is goal-directed, genes are switched on/off permanently
* CS: recursion = infinite loop, arrays are the only data structure
* Mathematics: correlation implies causation, division always makes smaller
Implementation: Retrieval-augmented lookup against a misconception knowledge base, indexed by concept × subject. The script generator is required to address the top-2 misconceptions for each concept node before proceeding.


1b. Scaffolding strategy selection
The agent selects an explanation structure before writing. The available strategies are:
* Example → generalization (best for concrete learners, younger audiences)
* Intuition → formula → application (best for IB/AP students comfortable with abstraction)
* Misconception → correction → reconstruction (best for topics with strong prior errors)
* Story → formalization (best for engagement-first personas, narrative learners)
Selection is parameterized by the student model from Layer 2. The strategy choice is logged per run so the feedback loop (Layer 3) can track which strategies win in the Elo arena.


1c. Cognitive load optimization
The script generator is constrained by measurable cognitive load proxies. The CIDPP critic enforces these limits:
Parameter
	Target range
	Enforcement
	Concept density
	≤ 2 new concepts per 3 minutes
	Critic flags violations, triggers split
	Equation-to-analogy ratio
	≤ 1:2 for non-calculus personas
	Persona parser sets ratio ceiling
	Average sentence length
	≤ 18 words for age < 16
	Script post-processor trims
	Quiz/check frequency
	1 check per 4–5 minutes
	Planner enforces checkpoints
	

1d. Explanation style adaptation
The agent does not just acknowledge the student persona — it reasons about implied prior knowledge. A persona string like 'High schooler, no calculus' implies: comfort with algebra, unfamiliarity with limits/derivatives, likely familiar with slope but not as a function. The persona parser extracts these implied constraints and the script generator respects them throughout.


Layer 2 — Student modeling
From the student_persona string, the agent constructs a structured student model object that drives all downstream decisions.


Student model schema
student_model = {
  level:                 "IB" | "AP" | "high_school" | "middle_school",
  knowledge_embedding:   [...],   // implied known concepts
  misconception_risk:    {...},   // top-3 likely errors by concept
  cognitive_load_budget: float,  // max concept density ceiling
  modality_preference:   "visual" | "verbal" | "mixed",
  abstraction_tolerance: float,  // 0=concrete only, 1=fully abstract
}
	

Zone of Proximal Development (ZPD) targeting
The concept planner (M3) must target the student's ZPD: starting just below their ceiling and building up. The lesson arc must not assume any concept without having introduced it earlier in the same video. The planner validates this by checking every node in the concept graph against the student's knowledge_embedding before finalising the order.


Synthetic student testing
Before committing to video generation, the agent runs 4 synthetic student personas against the draft script. Each persona role-plays the target student and reports: what was still confusing, what assumed knowledge they did not have, where pacing was too fast. Revisions are triggered if any persona flags more than 2 gaps.
* Persona A: Confused visual learner — needs diagrams before words
* Persona B: Math-anxious student — flags equations without analogies
* Persona C: High-performing abstract thinker — flags oversimplification
* Persona D: Low prior knowledge, high curiosity — flags missing prerequisites


Layer 3 — Teaching strategy learning loop
This is the layer that separates the winning team from everyone else. The agent treats every API call as a training event and updates its teaching policy from every signal available.


3a. Feedback signal collection
Three signal sources are collected and stored in the feedback logger (M8):
* AI Student scores (Phase 1): structured per-dimension scores and textual critique from the automated evaluator. Parsed into: which concept was unclear, where logic broke down, vocabulary mismatch. Treated as the primary training signal during warm-up.
* Elo outcomes (Phase 2): win/loss per match, opponent style (if observable), concept × persona context. Used to update the teaching policy and reward model.
* Expert review signals (Phase 3): qualitative feedback from professors. Stored as preference data for reward model fine-tuning.


3b. Teaching reward model
The agent maintains a lightweight reward model R(script, student_model) → float that predicts teaching effectiveness before the video is rendered. This enables Best-of-N selection: generate 3 script variants, score each, commit to the highest scorer.
The reward model follows the Reinforcement Learned Teacher (RLT) paradigm: the reward is not script fluency, but the predicted probability that a student model recovers the correct answer given the explanation. This aligns the teacher's objective with actual learning outcomes.
Initial implementation: a prompted LLM judge scoring on the CIDPP rubric (see M5). Over time, trained on accumulated preference data from AI Student reports and Elo outcomes.


3c. Self-play A/B evaluation
For each lesson, the script generator produces N=3 variants using different scaffolding strategies (Layer 1b). The reward model scores all three and selects the highest. The unchosen variants and their scores are logged as negative preference examples for future reward model training.


3d. Pedagogy memory bank
Every run that achieves an AI Student score ≥ 8/10 or an Elo win is stored as a few-shot example in the pedagogy memory bank, indexed by concept × student_level. Future runs on similar topics retrieve the top-2 stored examples and use them as positive demonstrations in the script generator's system prompt.
This creates a compounding advantage: every strong performance makes the next similar lesson better. By the time of the finals, the agent's memory bank contains accumulated best-practice pedagogical moves for every tested domain.


3e. Meta-policy for strategy selection
The feedback logger tracks win rate per scaffolding strategy × student_level × subject combination. Over time the agent builds a meta-policy:
* Physics + non-calculus → intuition-first + geometric analogy (tracked win rate)
* CS + high prior knowledge → misconception-correction approach (tracked win rate)
* Biology + visual learner → diagram-first narrative (tracked win rate)
The strategy selector (Layer 1b) uses this meta-policy to bias its choice on new requests, while still exploring undersampled strategies with ε-greedy probability (ε = 0.15 initially, decaying to 0.05 after 50 runs).

PCK analogy retrieval
The agent maintains a retrieval store of expert pedagogical moves — specific, high-quality analogies indexed by concept and student level. These are curated, not generated:
* Recursion (CS, beginner) → Russian nesting dolls; mirrors facing each other
* Conservation of momentum (Physics) → billiard balls; ice skater pulling arms in
* DNA transcription (Biology) → photocopying a blueprint without taking the original out
* Derivative (Maths, no calculus) → speedometer reading vs. distance on odometer
* Self-attention (CS/ML, high school) → highlighting every word in a sentence and asking: which other words help me understand this one?
The script generator queries this store before writing each concept segment. Retrieved analogies are inserted before any formal definition. New high-scoring analogies from AI Student feedback are added to the store post-run.


Narration pacing requirements
* Hook rate: The first 60 seconds must contain a curiosity trigger (open question, counterintuitive fact, or relatable scenario) before any formal content
* Summary beats: A micro-summary must appear every 4–5 minutes of content
* Socratic questions: Minimum 2 prediction prompts per lesson ('What do you think happens when...')
* Progressive reveal: Complex diagrams must be introduced element-by-element, not shown complete


Layer 5 — Self-improving evaluation system
The CIDPP critic (M5) runs on every draft script before video generation. It is the quality gate. No script proceeds to video rendering unless it passes the critic's minimum thresholds.

4. NotebookLM Integration
4.1 Role and boundaries
NotebookLM is integrated as the sourcing and fact-grounding module (M1). It does not replace any of the five capability layers — it provides verifiable, cited factual content that the teaching agent uses as the script's factual foundation.


Critical constraint: NotebookLM's video and audio generation features are NOT reliably exposed through the official API as of April 2026. Do NOT architect around NotebookLM generating the video. It handles sourcing only. Your video pipeline (M7) must be fully independent.

4.5 MCP integration failure handling
The 30-minute generation window cannot be held hostage to an upstream API. The following fallback chain must be implemented:
1. Attempt NotebookLM MCP sourcing (timeout: 90 seconds)
2. On timeout or error: fall back to web_search + web_fetch for authoritative sources
3. Run code interpreter verification on all formulas and numerical claims regardless of sourcing path
4. Log the failure mode in M8 for post-run analysis
The CIDPP critic's Integrity dimension (threshold: 9/10) serves as the final safety net regardless of which sourcing path was used.

9. Competitive Advantages and Research Directions
These are the highest-leverage differentiators, ordered by expected impact on Elo ranking.


1. RLT-style student-aligned reward (highest leverage)
The Sakana AI Reinforcement Learned Teacher paradigm is the most theoretically sound approach to optimising 'better teaching' as a measurable objective. The teacher model is given both the question and the correct answer; its reward is the student model's log-probability of recovering the correct answer given the teacher's explanation. This is not yet widely implemented in competition contexts. First-mover advantage is significant.
Implementation dependency: requires a student model (a smaller LLM used as the comprehension evaluator). Can be prototyped with a 7B parameter model. Recommended for Phase 3 / pre-finals implementation.


2. Persona-conditional PCK retrieval (medium leverage, low effort)
A growing knowledge base of expert pedagogical moves indexed by concept × learner_level is a compounding asset. Every competition run that scores highly adds entries. Competitors using generic prompting do not have this. Begin seeding in Week 1 with 5 entries per subject domain; target 100+ entries by finals.


3. Pipeline meta-optimization (medium leverage, medium effort)
Monitoring which module stage most often causes low CIDPP scores and dynamically allocating more compute or a better model to that stage is an engineering advantage not captured in any single model improvement. Example: if M3 concept sequencing is consistently flagged by the critic, replace M3's model with a stronger one while keeping M4 on a cheaper model.


4. Synthetic student diversity (medium leverage, low effort)
The more granular and realistic the 4 synthetic personas tested pre-evaluation, the better the revision signal. Calibrate each persona against real AI Student feedback from Phase 1 to make the synthetic test predictive of the real evaluator's concerns.


5. Few-shot pedagogy memory prompting (low leverage to implement, high compounding value)
Storing top-scoring lesson scripts as few-shot examples and prefixing them into the M4 system prompt for similar topics costs almost nothing to implement but compounds significantly over the competition. By the finals, the agent is not generating lessons from scratch — it is generating lessons that are variations on its own best prior work.

11. Glossary


Term
	Definition
	PCK
	Pedagogical Content Knowledge — knowing how to teach a subject, not just knowing the subject
	CIDPP
	Clarity, Integrity, Depth, Practicality, Pertinence — the 5-dimension rubric used by the CIDPP critic (M5)
	ZPD
	Zone of Proximal Development — the range just above what a student can do alone; the optimal teaching target
	RLT
	Reinforcement Learned Teacher — a training paradigm where the teacher's reward is the student's comprehension gain
	Elo
	The pairwise ranking system used in Phase 2 of the competition to rank agent outputs based on student preference
	MCP
	Model Context Protocol — the open standard used to connect agents to external tools like NotebookLM
	NLM
	NotebookLM — Google's research tool used in M1 for source grounding and supplementary material generation
	Best-of-N
	Generating N script variants and selecting the one with the highest reward model score
	Meta-policy
	The learned policy for selecting scaffolding strategies based on accumulated win-rate data per (strategy, level, subject)
	Pedagogy memory bank
	The retrieval store of highest-scoring past lessons used as few-shot examples in M4
